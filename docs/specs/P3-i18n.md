# P3 — UI 中英文切换 + LLM 输出双语 Spec

## 1. 目标

让用户在 UI 右上角切换 EN / 中文，**同时切换**：
1. 所有界面静态文案（按钮、标签、tooltip、错误提示）
2. LLM 输出的自然语言字段（rationale / risk / disconnection / 等）

**不翻译**的东西（它们是国际标准）：SMILES, InChI, IUPAC, 化学式, SMARTS, 元素符号, 反应类名（"Suzuki coupling" 保持原文）。

---

## 2. 设计

### 2.1 UI 文案（前端）

不引入 `next-intl`（依赖太重，且 SSR 复杂度高）。自写一个极简方案：

```typescript
// frontend/lib/i18n/types.ts
export type Lang = "en" | "zh";

// frontend/lib/i18n/en.ts
export const en = {
  "tab.fga": "Functional Groups",
  "tab.conditions": "Reaction Conditions",
  "tab.retro": "Retrosynthesis",
  "input.placeholder": "Paste SMILES, InChI, MOL block, or reaction A.B>R>C",
  "input.invalid": "Invalid input — please check format",
  "btn.analyze": "Analyze",
  "btn.analyzing": "Analyzing...",
  "confidence.high": "High confidence",
  "confidence.med": "Medium confidence",
  "confidence.low": "Low confidence",
  "retry.label": "Retried {{n}} time(s)",
  "ocr.not_verified": "OCR result not verified by round-trip",
  // ... 总计预计 ~40 条
} as const;

// frontend/lib/i18n/zh.ts —— 同形键，值替换
```

`zh` 必须是 `typeof en` 类型才能编译，强制保证翻译完整性。

```typescript
// frontend/lib/i18n/index.ts
const dicts = { en, zh };
export function t(lang: Lang, key: keyof typeof en, vars?: Record<string, string|number>): string {
  let s: string = dicts[lang][key] ?? en[key];
  if (vars) for (const [k, v] of Object.entries(vars)) s = s.replace(`{{${k}}}`, String(v));
  return s;
}
```

通过 `LanguageContext`（React Context）传 `lang` + `setLang`，挂在 `app/layout.tsx`，状态写入 `localStorage`，初值读 `navigator.language` 启发式检测。

### 2.2 LLM 输出语言（后端 + Prompt）

**Prompt 改动**：

`backend/app/pipeline/prompts/system.j2`（共享 header）开头追加：

```
Respond in {{ output_language_name }}.
For chemistry identifiers (SMILES, InChI, IUPAC names, molecular formulas, SMARTS, element symbols, named reactions like 'Suzuki coupling'), KEEP THEM IN ENGLISH / standard notation — do not translate them.
Only translate natural-language fields: rationale, risk, disconnection description, summary text.
```

`output_language_name` 是字符串 `"English"` 或 `"Chinese (Simplified, 简体中文)"`。

**API 改动**：

每个模块的 request schema 加 `lang: Literal["en", "zh"] = "en"`。FastAPI 端透传给 `prompts.render(..., output_language=lang, output_language_name=LANG_NAME[lang])`。

**缓存 key 改动**：

`backend/app/cache.py` 已有 `_key_for(...)` 函数，把 `lang` 加进 key 计算——中英文不共享缓存。

### 2.3 切换交互

- 顶部右上 `EN | 中` toggle（小按钮，不要 dropdown）
- 切换时：
  1. 立刻重渲染所有 UI 文案
  2. **不**自动重跑当前结果（结果已经是某语言生成的，不强制刷新；用户重跑时才用新语言）
  3. 在结果卡片 header 加一个小角标：`generated in EN` / `生成语言: 中文`，避免用户疑惑为何当前结果还是旧语言

## 3. 入口文件

| 文件 | 改动 |
|---|---|
| `frontend/lib/i18n/{types,en,zh,index}.ts` (新建) | 字典 + `t()` 函数 |
| `frontend/components/LanguageContext.tsx` (新建) | Context + Provider，localStorage 同步 |
| `frontend/components/LangToggle.tsx` (新建) | 顶部切换按钮 |
| `frontend/app/layout.tsx` | 挂 `<LanguageProvider>` |
| `frontend/components/**/*.tsx` (~8 个) | 所有硬编码字符串改 `t(lang, "key")` |
| `frontend/lib/api.ts` | 所有 POST 加 `lang` 字段 |
| `frontend/lib/types.ts` | 响应类型加 `output_language?: "en" | "zh"` |
| `backend/app/pipeline/prompts/system.j2` | 加语言指令 |
| `backend/app/pipeline/prompts/{fga,conditions,retro}.j2` | （如不 extends `system.j2`）各自补充一行语言指令 |
| `backend/app/api/{fga,conditions,retro}.py` | request schema 加 `lang` 字段；响应回写 `output_language` |
| `backend/app/modules/{fga,conditions,retro}.py` | 把 `lang` 传给 `prompts.render` |
| `backend/app/cache.py` | `_key_for` 把 `lang` 也哈希进去 |

## 4. 测试样例

### 4.1 前端单元（Jest 或 Vitest，可选）

`frontend/lib/i18n/__tests__/t.test.ts`

| 用例 | 期望 |
|---|---|
| `t("en", "tab.fga")` | `"Functional Groups"` |
| `t("zh", "tab.fga")` | `"官能团警报"`（或定稿翻译）|
| `t("zh", "retry.label", {n: 2})` | `"已重试 2 次"` |
| `t("zh", "missing.key" as any)` | fallback 回 en 值，不抛 |

**类型测试**：故意在 `zh.ts` 删掉一个 key，`tsc --noEmit` 应该报错。这是关键保证。

### 4.2 后端模块测试

`backend/tests/test_lang_propagation.py`（mock LLM）

| 用例 | 期望 |
|---|---|
| `test_fga_lang_en` | mock provider 收到的 system prompt **包含** `Respond in English` |
| `test_fga_lang_zh` | mock provider 收到的 system prompt **包含** `Respond in Chinese` |
| `test_fga_lang_default` | 不传 `lang` 参数 → 默认 en |
| `test_cache_lang_separation` | 同一 SMILES，先 `lang=en` 调用一次，再 `lang=zh` 调用一次 → mock 应被调用**两次**（缓存不命中） |
| `test_cache_lang_repeat_hits` | 同 SMILES + `lang=zh` 调用两次 → 第二次命中缓存，mock 被调用 1 次 |
| `test_invalid_lang_rejected` | `lang=fr` → 422 |

### 4.3 端到端（Playwright + 真实 LLM，`@slow`）

`frontend/tests/e2e/i18n.spec.ts`

| 用例 | 操作 | 期望 |
|---|---|---|
| `test_default_lang_en` | 首次打开 | UI 英文，输入框 placeholder 英文 |
| `test_toggle_to_zh` | 点"中" | 所有标签变中文 |
| `test_persist_after_reload` | 切中文 → 刷新页面 | 仍是中文 |
| `test_llm_output_zh` | 切中文 → 输入 aspirin → 跑 FGA | `risk` 字段是中文，但 `group` 仍是英文（如 `carboxylic_acid`），SMILES 保持英文 |
| `test_toggle_after_result` | 用 en 跑出结果 → 切 zh | UI 标签变中文，但结果卡片标 "生成语言: EN"，结果内容不变 |

### 4.4 LLM 双语质量检查（人工抽检，标记 `@manual`）

5 个 fixture 分子分别用 en/zh 跑一次，人工核对：
- 中文输出**没有**翻译 SMILES（不应出现 "C乙C" 这种）
- 反应类名保持英文（"Suzuki coupling" 不译成"铃木偶联"或同时给）
- IUPAC 保持英文 IUPAC 命名（不译成中文化学命名）
- 中文表述自然，无机翻味

## 5. 测试闭环

```
PR commit
  → CI
    → tsc --noEmit（强制翻译完整性，缺 key 直接 fail）
    → pytest -m "not slow"（4.2 模块测试 + cache 测试）
    → frontend Vitest（4.1，如有配）
  → 本地
    → pytest -m slow（4.3 E2E + 真实 LLM）
    → 人工抽检 4.4
```

**绿灯条件**：
1. tsc 编译 0 错（保证翻译完整性）
2. cache lang 分离测试通过（保证不串语言）
3. 5/5 人工抽检通过化学名词不被翻译

---

## 6. 验收标准

- [ ] `tsc --noEmit` 全绿（任一中文 key 缺失即 fail）
- [ ] 后端 6 个 lang 相关单测通过
- [ ] UI 切换 5 次无状态错乱
- [ ] 中文输出 0 处错误翻译化学标识（5 用例人工抽检）
- [ ] Cache lang 分离已验证（同 SMILES en/zh 各跑一次后查 DB 有两行）
- [ ] 已存结果切换语言时显示 "generated in EN/ZH" 角标

**预估工时**：1–1.5 天（翻译稿耗时一半，工程一半）
